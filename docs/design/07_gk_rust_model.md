# GK Rust Implementation Model

How the GK design maps to Rust types, traits, and runtime structures.

---

## Implementation Strategy

The implementation proceeds in two phases:

1. **Phase 1 (now): Runtime graph** — the assembled DAG is a runtime
   data structure with dynamic dispatch at node boundaries. Nodes carry
   metadata (port names, types, arity) that enables correct wiring and
   validation at assembly time. This gets us to a working system quickly.

2. **Phase 2 (later): Compiled graph** — the runtime graph is strict
   enough to serve as input to ahead-of-time compilation (via proc
   macros, cranelift, or similar). The same metadata that validates
   wiring at runtime can drive monomorphized code generation that
   eliminates dynamic dispatch on the hot path.

The key constraint: Phase 1's runtime model must carry enough type and
signature information that Phase 2 can consume it without redesign.

---

## Macro System for Variadic Signatures

Rust macros will generate the boilerplate for declaring node functions
with strict, statically typed signatures. A single macro invocation
should produce:

- The node function implementation
- Runtime metadata (port names, types, arity) for assembly-time wiring
- Trait implementations for the node evaluation interface

### Sketch

```rust
// Macro declares a node with typed, named ports
gk_node! {
    /// Decompose a value into mixed-radix digits
    fn mixed_radix(value: u64, #[params] radixes: Vec<u64>)
        -> (Vec<u64>)  // output port count determined by radixes.len()
    {
        // implementation
    }
}

gk_node! {
    /// Murmur3-style hash
    fn hash(input: u64) -> (u64) {
        // implementation
    }
}

gk_node! {
    /// Combine two u64 values by interleaving bits
    fn interleave(a: u64, b: u64) -> (u64) {
        // implementation
    }
}

gk_node! {
    /// Sample from a normal distribution given uniform input
    fn normal(uniform_input: u64, #[param] mean: f64, #[param] stddev: f64)
        -> (f64)
    {
        // implementation
    }
}
```

### What the macro generates

For each `gk_node!` invocation:

1. **A struct** implementing the node's logic and holding assembly-time
   parameters.

2. **A `NodeMeta` descriptor** available at runtime:
   ```rust
   NodeMeta {
       name: "mixed_radix",
       inputs: &[Port { name: "value", typ: PortType::U64 }],
       outputs: &[Port { name: "out", typ: PortType::U64 }],  // or dynamic
       params: &[Param { name: "radixes", typ: ParamType::VecU64 }],
   }
   ```

3. **Trait impl** for the node evaluation interface (see below).

### Annotation attributes

- `#[param]` — an assembly-time constant, not a wire input. Baked into
  the node at construction, not supplied per-cycle.
- `#[params]` — a variadic assembly-time parameter (e.g., the radix
  list for `mixed_radix`) that may influence the output port count.
- Bare parameters (no attribute) — wire inputs, connected to upstream
  nodes or coordinates.

## Runtime Metadata

Every node instance carries metadata sufficient for the assembly phase
to validate wiring without executing any node logic. This metadata is
also the bridge to Phase 2 compilation.

### Port Types

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum PortType {
    U64,
    F64,
    Bool,
    String,
    Bytes,
    // Extensible for Phase 2
}
```

### Port Descriptor

```rust
#[derive(Debug, Clone)]
pub struct Port {
    pub name: &'static str,
    pub typ: PortType,
}
```

### Node Metadata

```rust
#[derive(Debug, Clone)]
pub struct NodeMeta {
    pub name: &'static str,
    pub inputs: Vec<Port>,
    pub outputs: Vec<Port>,
    pub params: Vec<Param>,
}
```

This metadata is generated by the macro and queryable at runtime. The
assembly phase uses it to:
- Validate that wired ports have compatible types
- Determine where to auto-insert edge adapters
- Report type errors with meaningful names
- (Phase 2) Generate monomorphized evaluation code

## Node Evaluation Trait

The runtime trait for evaluating a node. Phase 1 uses dynamic dispatch
via trait objects; Phase 2 replaces this with generated code.

```rust
/// A value flowing through the DAG at runtime
#[derive(Debug, Clone)]
pub enum Value {
    U64(u64),
    F64(f64),
    Bool(bool),
    Str(String),
    Bytes(Vec<u8>),
}

/// Runtime evaluation interface for a GK node
pub trait GkNode: Send + Sync {
    /// Return this node's metadata (ports, types, params)
    fn meta(&self) -> &NodeMeta;

    /// Evaluate the node given its input values, write to output slots
    fn eval(&self, inputs: &[Value], outputs: &mut [Value]);
}
```

### Phase 1 characteristics

- `inputs` and `outputs` are slices of `Value` enum — dynamic typing
  at the value level, but the assembly phase has already validated that
  the types are correct.
- `eval` is called via `dyn GkNode` — one virtual dispatch per node
  per pull.
- This is acceptable for Phase 1. The overhead is small relative to
  the cost of the operations themselves (hashing, distribution sampling,
  string building, CSV lookup).

### Phase 2 path

The same `NodeMeta` that drives runtime wiring can drive compile-time
code generation:
- Proc macros or build scripts consume the metadata to generate a
  monomorphized evaluation function for the entire DAG.
- The `Value` enum is eliminated — each wire becomes a typed local
  variable.
- Virtual dispatch is eliminated — each node call is a direct function
  call.
- The generated code is a single function:
  `fn eval(coords: &[u64], outputs: &mut OutputStruct)`.

## DAG Runtime Structure

```rust
pub struct GkKernel {
    /// Node instances in topological order
    nodes: Vec<Box<dyn GkNode>>,

    /// Wiring table: for each node input port, which (node_index,
    /// output_port_index) provides its value
    wiring: Vec<Vec<WireSource>>,

    /// Coordinate input names and their positions
    coord_names: Vec<String>,

    /// Output variate names and which (node_index, output_port_index)
    /// provides each
    output_map: HashMap<String, (usize, usize)>,

    /// Per-node output value buffers (reused across evaluations)
    buffers: Vec<Vec<Value>>,
}

pub enum WireSource {
    Coordinate(usize),          // index into coord tuple
    NodeOutput(usize, usize),   // (node_index, port_index)
}
```


### Coordinate context and memoization

Each `pull` call within the same coordinate context can reuse
previously computed node outputs. A simple generation counter tracks
the current context:

```rust
pub struct GkKernel {
    // ... fields above ...
    generation: u64,
    node_generation: Vec<u64>,  // last-evaluated generation per node
}
```

When `generation` advances (new coordinate), all cached values are
implicitly invalidated. A node is only re-evaluated if its
`node_generation` differs from the current `generation`.

## Edge Adapter Implementation

Edge adapters are just nodes with 1→1 signatures. They are
auto-generated during assembly and inserted into the `nodes` array
like any other node.

```rust
gk_node! {
    fn __u64_to_string(input: u64) -> (String) {
        input.to_string()
    }
}

gk_node! {
    fn __u64_to_f64(input: u64) -> (f64) {
        input as f64
    }
}

gk_node! {
    fn __f64_to_string(input: f64) -> (String) {
        input.to_string()
    }
}
```

In the kernel IR, they appear as regular nodes. The distinction between
"edge adapter" and "domain node" exists only in the surface DSL and
assembly phase — the runtime doesn't care.

## Crate Structure (Sketch)

```
nb-variates/
├── src/
│   ├── lib.rs              // Public API: GkKernel, pull interface
│   ├── node.rs             // GkNode trait, NodeMeta, Port, Value
│   ├── kernel.rs           // GkKernel runtime structure, evaluation
│   ├── assembly.rs         // Assembly pipeline (wiring, validation)
│   ├── dsl/
│   │   ├── mod.rs          // DSL parser entry point
│   │   ├── lexer.rs        // Tokenization
│   │   ├── parser.rs       // AST construction
│   │   └── desugar.rs      // Sugar normalization to kernel IR
│   ├── nodes/
│   │   ├── mod.rs          // Node registry
│   │   ├── hash.rs         // Hash64, HashRange, HashInterval
│   │   ├── arithmetic.rs   // Add, Mul, Div, Mod, Clamp, Min, Max
│   │   ├── decompose.rs    // MixedRadix, Unpack, CycleRange
│   │   ├── compose.rs      // Interleave, Concat, Pack
│   │   ├── distribute.rs   // Normal, Zipf, Poisson, etc.
│   │   ├── lookup.rs       // WeightedLookup (CSV-backed)
│   │   ├── string.rs       // Template/str_build, Join, Prefix, etc.
│   │   ├── convert.rs      // Edge adapters (u64→String, etc.)
│   │   ├── identity.rs     // Identity, Const, ConstStr
│   │   └── time.rs         // EpochMillis, ToTimestamp, ToUUID

├── tests/
│   ├── dsl_parse.rs        // DSL parsing tests
│   ├── assembly.rs         // Wiring validation tests
│   ├── eval.rs             // End-to-end evaluation tests
│   └── nodes/              // Per-node-category tests
└── Cargo.toml
```
