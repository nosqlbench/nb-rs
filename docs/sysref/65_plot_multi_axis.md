# SRD-65 — Plot multi-axis support (`y`, `y2`, `y3`, …, `yN`)

**Status:** shipped (axis 3+ rail-tick rendering deferred — see Followups)
**Owner:** runtime / nbrs (plot_metrics)
**Implementation:** `nbrs/src/plot_metrics.rs` (renderer),
  `nbrs-workload/src/report/vocab.rs` (directive vocab,
  per SRD-46v2)
**Cross-refs:** SRD-46 (reports), SRD-64 (report CLI),
  SRD-47 (MetricsQL streaming — supplies query rows)

---

## Why

Today a plot has a primary y-axis (`y:`) and one optional
secondary y-axis (`y2:`). The two-axis surface covers the
common "metric + companion metric on a different scale"
case (recall + overscan, throughput + latency) but stops
there. Real reports want more:

- recall + p50 latency + overscan — three independent scales
- throughput + p50 + p99 + p999 — four percentile rails
- any composition of metrics whose y-units don't line up

The grammar can grow to support N axes mechanically; the
renderer is the load-bearing part because plotters' built-in
`set_secondary_coord` is a one-shot op (linear/log + range,
exactly one secondary axis). N≥3 needs us to project each
axis's data into the primary coordinate space and draw axis
rails + labels manually.

---

## Naming

The directive prefix for axis *N* is `yN`:

| Axis | Directive prefix | Notes |
| --- | --- | --- |
| 1 (primary) | `y` or `y1` | both refer to the primary axis. `y1` is preferred when the operator wants the directive list to read uniformly across axes (`y1:`, `y2:`, `y3:`, `y4:`). |
| 2 (first secondary) | `y2` | unchanged from today |
| 3 …        | `y3`, `y4` | new |

`y` and `y1` are accepted as synonyms for axis 1 — a
deliberate exception to the "no aliases" rule because
operators routinely want both the bare-`y` ergonomics
(simple plots) and the uniform `y1`/`y2`/`y3`/`y4` listing
(multi-axis plots). Mixing forms inside a single plot body
(`y:` *and* `y1:`) is rejected as ambiguous.

Axis indices must be contiguous: a plot with `y3:` but no
`y2:` is a hard error. This prevents "skipped axis" sparse
configurations that would be ambiguous to lay out.

The maximum axis count is **4** (`y` + `y2` + `y3` + `y4`).
Beyond that the chart becomes unreadable; the recommended
pattern is faceting (small multiples), which is a separate
work item parked in SRD-98.

---

## Per-axis directive set

Every axis carries the same directive surface as the primary
today. For axis prefix `yN` (where `yN` may be `y` for axis 1
or `y2`/`y3`/`y4` for the rest):

| Directive          | Purpose |
| ---                | --- |
| `yN:` *(query)*    | MetricsQL expression that supplies this axis's series |
| `yN-label:`        | axis title (right-side rail label) |
| `yN-min:`          | lower bound (number or `auto`) |
| `yN-max:`          | upper bound (number or `auto`) |
| `yN-range:`        | `a..b` / `(a, b)` / `[a, b]` shorthand for min+max |
| `yN-scale:`        | `linear` / `log` / `auto` (auto runs the tick-shape classifier) |
| `yN-ticks:`        | literal CSV / `auto` / metricsql expression for explicit detents |
| `yN-legend:`       | per-series legend-label template; `[label_name]` placeholders are substituted with the series's discriminator-label values at render time |

Mutual-exclusion rules already enforced by
`AxisDirectiveTracker` (SRD-46v2 / current code) extend
verbatim: `yN-range` conflicts with `yN-min` / `yN-max` on
the same axis. Different axes are independent.

For axis 1 only, the directive prefix is `y` rather than
`y1` — i.e. `y-min:`, `y-range:`, etc. — so existing plot
specs don't change. The dispatch table is generated by
expanding `[("y", 1), ("y2", 2), ("y3", 3), ("y4", 4)]`
across the seven directive forms above.

## Per-series legend templating

`yN-legend: "<template>"` overrides the auto-generated
series-name shape (`profile=label_03, k=10`) with a
caller-supplied template. Bracketed placeholders
(`[label_name]`) are substituted with that series's
corresponding discriminator-label value at render time:

```yaml
plot recall_curves
  y: avg(recall_at_1_mean{k="1"}) by (profile, limit)
  x: limit
  y-legend: "oracle-[profile]"        # → "oracle-label_03"
  y2: avg(overscan{k="1"}) by (k, limit)
  y2-legend: "overscan k=[k]"         # → "overscan k=1"
```

Rules:
- Match misses (`[unknown]`) are left verbatim — operators
  see their typo rather than a silent empty substring.
- Unterminated `[` is treated as a literal `[`.
- The template fully replaces the auto-label, including the
  per-axis suffix (`(R)`, `(R2)`). If the operator wants
  axis disambiguation in the legend, they include it
  themselves: `y2-legend: "[k] (R)"`.
- Surrounding quotes (`"..."` or `'...'`) on the directive
  value are stripped — operators tend to write
  `y2-legend: "oracle-[profile]"` whether YAML requires
  the quotes or not.

Storage: `AxisOpts.legend_format: Option<String>` for axes
2..N; `PlotMetricsOpts.y_legend_format: Option<String>` for
axis 1 (which lives outside `secondary_axes`).

## Per-series style overrides across axes

`style <key>=<value>:<directives>` matches series by their
discriminator labels (auto-detected per axis as of the y2
auto-detect fix). The match is axis-agnostic — a `style
profile=label_03:line=dashed` directive applies to whichever
axis (`y` / `y2` / `y3` / …) has a series tagged
`profile=label_03`. The renderer walks each axis's series
list and applies the first matching override.

This keeps the override grammar simple at the cost of a
collision case: if two axes happen to have a series with the
same discriminator, one override applies to both. That's the
right default — overlapping discriminators across axes are
themselves a smell — but if it bites, an axis-scoped form
(`style:y2 key=value:...`) can be added later. Out of scope
for this SRD.

## Layout

The chart frame:

```
   ┌────────────────────── caption ──────────────────────┐
   │                                                     │
y  │                                                     │  y2
-> │              [ plot drawing area ]                  │ <-
   │                                                     │  y3
   │                                                     │ <-
   └─────────────── x ───────────────────────────────────┘
```

- **Axis 1 (`y`)** lives on the left edge, as today.
- **Axis 2..N (`y2`, `y3`, `y4`)** stack on the right edge,
  moving outward. Each axis adds 70px to
  `right_y_label_area_size` (the same default used for `y2`
  today). Axis labels rotate 90° on the rail; tick labels
  read horizontally.
- The axis labels (right-rail titles) are color-coded to
  match a representative series on that axis — picked as the
  axis's first series in canonical order, or the override
  color if any series-style override pins one.
- A small color swatch beside each axis label disambiguates
  when more than one series shares the axis.
- Per-axis tick labels render verbatim (same `F64Axis`
  formatter as today) on each rail.

Rendering order:

1. Frame + primary mesh (uses primary x + primary y).
2. y series drawn against primary coord.
3. For each `y_k` (`k = 2..N`): project the data into the
   primary coordinate space, draw the series, and stamp the
   right-rail axis (label + tick text + per-axis tick marks).

Step 3 doesn't use `set_secondary_coord` past `k=2` —
plotters' chart context only has one secondary slot. For
axis 3+ we manually:

- Compute a coordinate transform `f_k: yN_value → primary_y`
  using the axis's resolved range (linear or log) and the
  primary axis's resolved range as the target.
- Apply `f_k` to each datapoint before drawing.
- Render tick marks + labels on the rail at primary_y
  positions corresponding to the axis's natural detents
  (transformed through `f_k`).

Axis 2 keeps using `set_secondary_coord` today's path so the
existing rendering stays byte-identical when no axis 3+ is
declared. Adding axis 3+ rebuilds axes 2..N through the
projection path so all stacked rails come from one
mechanism.

## Legend

Series labels gain an axis suffix. The current `(R)` for y2
becomes part of a per-axis suffix vocabulary:

- axis 1: no suffix (today's behavior).
- axis 2: `(R)` (matches existing y2 plots — unchanged).
- axis 3: `(R2)`.
- axis 4: `(R3)`.

The suffix increments past axis 2 because operators expect
the *first* right-rail to read `(R)` (legacy ergonomics);
beyond that, numeric disambiguation is enough.

The per-axis label color in the legend matches the rail
swatch so `(R2)` and the y3 rail swatch line up visually.

## Auto-detect interactions

The recently-landed auto-detect paths apply per-axis without
new policy:

- **Series labels** — each axis runs
  `auto_detect_series_labels` against its own
  `rows_via_metricsql` result. Cross-axis labels are
  independent (already true for y/y2 after the fknn fix).
- **Scale (linear/log)** — `resolve_scale` runs on `yN-ticks`
  resolved against `yN`'s primary db. Each axis's
  `detect_scale_from_ticks` decision is independent.
- **Padding** — log axes use multiplicative padding, linear
  axes additive. Applied per-axis with the existing
  `pad_lo` / `pad_hi` helpers. No change.

The X-axis is shared across all y-axes (one chart, one X
range, one X tick set). All axes' rows participate in the
X-range computation — same as today's y/y2 rule.

## Failure model

Existing rules carry through; new ones called out:

- **Empty axis result** — if `yN:`'s metricsql returns no
  rows, the whole plot fails (per current y/y2 behavior).
  Errors propagate through the per-figure failure path
  (SRD-46 + the figure-error refactor) — no silent skip.
- **Sparse axis index** — `y3:` without `y2:` errors at
  parse time. The error names both the missing prefix and
  the "axes must be contiguous" rule.
- **Range conflict on an axis** — already handled by
  `AxisDirectiveTracker`; extend its key from
  `(AxisKey, AxisRole)` where `AxisKey ∈ {X, Y, Y2}` to
  `AxisKey ∈ {X, Y, Y2, Y3, Y4}`. No semantic change — the
  enum just grows.

---

## Migration

**No breaking change.** Existing y/y2 plots render
byte-identically because:

1. The directive vocab change is purely additive (adds y3 /
   y4 forms; doesn't change y / y2).
2. Axis 1 (`y`) keeps its left-edge placement.
3. Axis 2 (`y2`) keeps its right-edge placement and the
   `(R)` suffix.
4. The plotters `set_secondary_coord` path is reused when
   only y2 is present, so today's bitmap output is
   reproducible from the same plot spec.

The renderer rewrite expands one branch (manual projection
for y3+); existing branches are untouched.

## Implementation outline

Phasing — each phase is one PR-sized change:

1. **Vocab and parsing** — extend `parse_spec` /
   long-flag / key=value parsers and
   `FLAGS_TAKING_VALUE` with `y3*` and `y4*` directive
   families. Generate the dispatch table from the
   `[(prefix, idx), ...]` tuple list.
   - Extend `AxisKey` to `{X, Y, Y2, Y3, Y4}`.
   - Add per-axis fields to `PlotMetricsOpts`:
     `yN_query`, `yN_label`, `yN_min`, `yN_max`,
     `yN_scale`, `yN_ticks`. Or a `Vec<AxisOpts>`
     indexed by axis number — see "Open question" below.
   - Round-trip tests for each new directive form.

2. **Per-axis pipeline** — extract today's y2
   bucket→aggregate→range path into a function callable
   per axis. Run it for every declared `yN:`. No
   rendering changes yet — the data side is multi-axis,
   the chart side still only knows about y/y2.

3. **Renderer** — introduce the projection-based path:
   - Build axis transforms `f_k: yN → primary_y`
     (linear or log + range pair).
   - For axis 2: keep the `set_secondary_coord` path.
   - For axis 3+: project data, draw series via
     `chart.draw_series` (against primary coord), draw
     rail manually (text labels + tick marks at the
     right edge offset by `(k-2) * 70px`).
   - Rail label color matches the axis's representative
     series.
   - Legend suffixes per-axis (`(R)`, `(R2)`, `(R3)`).

4. **Tests** — add a 3-axis and a 4-axis plot to a fixture
   workload (or a dedicated test file) and golden-image
   compare the output. Image-level tolerance is fine;
   pixel-perfect isn't required.

5. **Docs + examples** — update `full_cql_vector.yaml` and
   the SRD-46 examples to demo a 3-axis plot. Add a
   troubleshooting note for the "sparse axis" error.

## Followups (not in initial implementation)

- **Rail-tick rendering for axis 3+.** Today axis 2 gets
  the full plotters native right-rail (axis description +
  tick labels + tick marks). Axes 3 and 4 are *projected*
  into the primary y-coord and drawn against it; their
  series appear at the right relative position but the
  chart has no visible rail showing what numeric value
  each y3/y4 line corresponds to. The legend suffix
  (`(R2)`, `(R3)`) plus the per-axis `yN-label:` are how
  the operator reads value attribution today.
  When this becomes a real pain point, draw the rails
  manually: extend `right_y_label_area_size` per extra
  axis, then after `chart.plotting_area().get_pixel_range()`
  is known, blit text labels and short tick marks at the
  rail x-position offsets. Out of scope for the initial
  push because a single-rail-only chart still tells the
  shape-of-the-data story.

- **Axis-scoped style overrides.** See SRD-98 entry; not
  implemented now because no workload has hit the
  cross-axis discriminator-collision case yet.

## Open questions

- **Storage shape — flat fields or `Vec<AxisOpts>`?**
  Flat fields (`y_query`, `y2_query`, `y3_query`, …) are
  what we have today and parse cleanly. A `Vec<AxisOpts>`
  is more uniform but loses the "axis 1 is the primary"
  asymmetry that the layout depends on. **Recommendation:**
  flat fields — `axes: Vec<AxisOpts>` indexed `0..N`
  internally, but exposed/parsed as named directives. Best
  of both: clean iteration in the renderer, no API
  churn at the directive layer.

- **Axis-scoped style overrides?** As noted in
  "Per-series style overrides", today's grammar
  is axis-agnostic. If a real workload hits the
  collision case, the extension is `style:yN
  key=value:directives`. Not in this SRD's scope — track
  in SRD-98.

- **Color budget.** With 4 axes, each potentially carrying
  N≥10 series, palette pressure goes up. The `wong`
  palette has 8 colors; `viridis` has more. Should the
  default palette pick adapt to total-series count?
  **Recommendation:** pick palette per workload as today;
  surface a warning when total series count exceeds the
  active palette's color count (informational, not an
  error).

- **Layout tunable?** `right_y_label_area_size` per axis
  is currently hard-coded at 70px. For dense charts the
  operator might want to tighten it. Out of scope for this
  SRD — handle as a `--axis-rail-width` flag if a real
  workload needs it.
